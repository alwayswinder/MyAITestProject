一、蓝图功能概述
--------------------------------------------------------------------------------
BP_Birds 是一个环境氛围鸟群模拟器，负责在游戏场景中呈现鸟群飞翔效果。
其核心功能包括：

1. 粒子系统驱动的鸟群视觉效果
   - 使用 Niagara 粒子系统（NiagaraComp）渲染鸟群外观
   - 可动态设置粒子数量（Set Niagara Variable Int32）和飞行状态（Set Niagara Variable Bool）
   - 通过 NiagaraFly 资产引用挂载对应 Niagara 系统

2. 路径点驱动的运动系统
   - 维护 PositionArray（Vector 数组）存储飞行路径点
   - 通过 CurrentPoint 索引追踪当前目标位置
   - 使用 VInterp to Constant 对 Actor 位置进行平滑插值，按 Speed 速度移动
   - 支持循环（bLoop=true）与单次（bLoop=false）两种飞行模式
   - 支持可选的 SplineUse 样条路径模式（通过外部样条 Actor 引用）

3. 玩家触发机制
   - 自定义事件 OnPlayerTrigger：响应碰撞重叠，检测玩家进入 TriggerRadius 范围
   - 触发后启动鸟群飞离动画，通过 bStart 标志位控制运动流程
   - ActiveCollision / CollosionStart / VisiableCollision 多状态机管理碰撞检测生命周期

4. 画质自适应（Scalability）系统
   - 每帧读取 GetGameUserSettings → GetOverallScalabilityLevel 画质等级
   - 根据画质等级动态调整 MaxNums（最大鸟群数量）和 DisNear（近距离阈值）
   - 使用 MapRangeClamped 将画质等级映射为具体数值
   - 缓存 lastScalaLevel 避免重复处理（但实现方式需要改进）

5. Timeline 动画
   - 单条 Timeline（NewTrack_0 浮点轨道）驱动飞行动画曲线
   - 配合 StartNoLoop 自定义事件触发单次飞行播放

6. 调试可视化
   - 包含 Draw Debug Sphere 节点用于开发期碰撞/路径可视化

7. 编辑器辅助
   - 包含 BillboardComponent，便于在编辑器中识别 Actor 位置

二、优化建议（按优先级排序）
--------------------------------------------------------------------------------

【P0 - 严重性能问题，必须修复】

▶ 优化项 1：Event Tick 逻辑过重（最高优先级）
  问题描述：
    EventGraph 共有 291 个节点，全部集中在单一图表中，且大量逻辑位于
    Event Tick 链路上。每帧执行：距离计算、数组访问、玩家引用获取、
    Niagara 变量设置、画质等级读取、Actor 位置更新等多项操作，
    造成严重的 CPU 帧预算浪费。
  优化方案：
    - 将不需要每帧执行的逻辑改用 Timer（SetTimer by Function Name / Event）驱动：
        * 画质等级检测：每 3~5 秒检查一次即可
        * 玩家距离检测（在未触发状态下）：每 0.2~0.5 秒一次即可
    - 仅保留真正需要每帧更新的逻辑在 Tick 中（如位置插值 VInterp）
    - 在 bStart=false 时完全关闭 Tick：调用 SetActorTickEnabled(false)，
      触发后再开启，可节省未激活状态下的所有 Tick 开销

▶ 优化项 2：Get Player Character 每帧调用
  问题描述：
    Get Player Character 节点位于 y≈3698，处于 Event Tick / OnPlayerTrigger
    链路上。每帧调用该函数会触发全局 PlayerController 查询，开销不可忽视。
  优化方案：
    - 在 Event BeginPlay 中缓存玩家引用到成员变量（如 CachedPlayer）
    - Tick/触发逻辑中直接使用缓存引用，避免重复查询
    - 若担心玩家重生/切换，可在 OnPlayerTrigger 时刷新缓存即可

▶ 优化项 3：GetGameUserSettings + GetOverallScalabilityLevel 每帧调用
  问题描述：
    画质等级查询（节点位置 y≈12001）每帧执行，而画质设置极少变更，
    完全没有必要每帧轮询。
  优化方案：
    - 在 Event BeginPlay 执行一次初始化
    - 绑定 GameUserSettings 的 OnSettingsChanged 委托监听变更事件
    - 或使用 SetTimerByFunctionName 每 5 秒检查一次
    - 与 lastScalaLevel 比较，仅在等级发生变化时更新 MaxNums / DisNear

--------------------------------------------------------------------------------

【P1 - 代码质量问题，建议修复】

▶ 优化项 4：EventGraph 过于庞大（291 节点 + 74+ 路由节点）
  问题描述：
    所有逻辑堆积在单一 EventGraph 中，包含至少 74 个 Reroute/Knot 节点
    （Knot_0 至 Knot_83），形成"蜘蛛网"式连线，极难维护和调试。
  优化方案：
    将功能拆分为独立函数（纯函数/非纯函数）：
      * UpdateMovement()       —— 处理路径点移动和位置插值
      * CheckPlayerProximity() —— 检测玩家距离并触发飞行
      * ApplyScalabilitySettings() —— 根据画质调整参数
      * InitializePositionArray()  —— BeginPlay 时初始化路径
      * UpdateNiagaraState()   —— 同步 Niagara 粒子状态
    减少 Reroute 节点，通过局部变量传递中间值

▶ 优化项 5：变量命名拼写错误
  问题描述：
    - CollosionStart（多写了一个 'l'）→ 应为 CollisionStart
    - VisiableCollision（多余字母）→ 应为 VisibleCollision
    这两处拼写错误会造成团队协作混淆，且在蓝图继承/接口对接时可能引发错误。
  优化方案：
    使用蓝图变量重命名功能修正拼写，并全局搜索引用确保无遗漏

▶ 优化项 6：数组操作频繁（PositionArray 访问）
  问题描述：
    PositionArray 的 Length 计算、Get (a copy)（深拷贝）以及 For Each Loop
    在 Tick 路径中反复执行，数组深拷贝在 UE Blueprint 中有额外内存开销。
  优化方案：
    - 在 BeginPlay 时缓存 PositionArray 长度到整型变量 PositionCount
    - 仅在 CurrentPoint 变化时读取对应位置，避免每帧 Get (a copy)
    - 如有 For Each Loop 在 Tick 中运行（用于 Debug 球体），移至非 Tick 路径

--------------------------------------------------------------------------------

【P2 - 次要问题，视情况处理】

▶ 优化项 7：Draw Debug Sphere 存在于生产图表
  问题描述：
    Draw Debug Sphere 节点（位于 y≈6476）未确认是否有 Development-Only 保护。
    若在 Shipping 版本中执行，每帧渲染调试球体会造成不必要的渲染开销。
  优化方案：
    - 用 "Is With Editor" / 宏 "Development Only" 包裹此节点
    - 或改用 UE_LOG 日志代替，Ship 构建自动剥离
    - 长期建议：调试完成后直接删除此节点

▶ 优化项 8：Niagara 变量每帧设置（无变化检测）
  问题描述：
    Set Niagara Variable (Bool) 和 Set Niagara Variable (Int32) 可能每帧调用，
    即使参数值未发生变化也会触发 Niagara 系统内部更新。
  优化方案：
    - 为需要同步的 Niagara 变量各添加一个"上次设置值"缓存变量
    - 在设置前与缓存比对，仅在值发生变化时才调用 Set Niagara Variable
    - 这可以消除大量不必要的 Niagara GPU 参数更新

▶ 优化项 9：SplineUse 变量存在但无对应 Spline 组件
  问题描述：
    变量 SplineUse（布尔）暗示支持样条路径模式，但蓝图组件列表中不包含
    SplineComponent，可能依赖外部 Actor。若此功能已弃用，该变量为死代码。
  优化方案：
    - 若功能有效：添加注释说明外部样条 Actor 的引用机制
    - 若功能弃用：移除 SplineUse 变量及相关分支逻辑，减少节点数量

▶ 优化项 10：Nums 与 MaxNums 关系不明确
  问题描述：
    同时存在 Nums（设计时数量）和 MaxNums（运行时根据画质调整的上限），
    两者交互逻辑未在变量描述中说明，易引起混淆。
  优化方案：
    - 为两个变量添加 Tooltip 注释，明确：
        Nums = 设计期望数量（编辑器配置）
        MaxNums = 运行时实际允许最大数量（由 Scalability 系统写入）
    - 在 Niagara 参数设置处，始终取 Min(Nums, MaxNums) 作为实际粒子数

▶ 优化项 11：缺乏 Event BeginPlay 显式初始化
  问题描述：
    在节点列表中未发现明确的 Event BeginPlay 节点，初始化逻辑可能散布于
    其他事件或 Tick 首帧中，导致初始化时序不可预测。
  优化方案：
    - 添加 Event BeginPlay，集中处理：
        * 缓存玩家引用
        * 执行首次画质等级检测
        * 初始化 PositionArray（若动态生成）
        * 设置 Niagara 初始状态
        * 如无需 Tick，初始时禁用 SetActorTickEnabled(false)

三、总结
--------------------------------------------------------------------------------
BP_Birds 整体功能完整，实现了鸟群生成、路径巡逻、玩家响应和画质自适应
等核心逻辑。主要性能风险集中在 Event Tick 中的过重逻辑链路，尤其是
GetPlayerCharacter 和 GetGameUserSettings 的每帧轮询。

优先修复顺序建议：
  P0 → P1 → P2

预期收益：
  - P0 修复后，每帧 CPU 开销预计降低 40%~60%（取决于场景中 BP_Birds 实例数量）
  - P1 修复后，代码可维护性显著提升，便于后续迭代
  - P2 修复后，进一步消除边缘开销，提升代码健壮性

节点统计：
  - EventGraph 总节点数：291
  - Reroute（Knot）节点数：≥74
  - 自定义事件数：3（OnPlayerTrigger, StartNoLoop, 及图中其他未完全枚举的事件）
  - 变量数：20
