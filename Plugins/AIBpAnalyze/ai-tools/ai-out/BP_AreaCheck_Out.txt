一、蓝图功能概述 | Overview
--------------------------------------------------------------------------------
BP_AreaCheck 是一个环境区域特效管理蓝图，核心功能为：
  - 利用样条（Spline）定义的多边形区域，通过射线交叉计数算法（Ray Casting）
    检测玩家角色是否进入指定区域。
  - 玩家进入区域后，在玩家身上附加 Niagara 粒子特效（区分昼夜：
    NS_Env_Common_PlayerVFXDay_LoopCom / NS_Env_Common_PlayerVFXNight_LoopCom）。
  - 支持昼夜切换时动态替换特效。
  - 支持图形质量分级（Scalability）检测，低品质等级下自动禁用特效。
  - 通过多个 Timer 控制特效的播放/暂停周期。
  - 玩家离开区域或Blueprint销毁时清理并移除所有特效。

组件结构 (SCS Components):
  - SceneComponent (根组件)
    └── BillboardComponent (编辑器可视化图标, 缩放 5×5×5)

引用依赖:
  - BP_SplineEditorOnlyLoop          样条区域边界定义
  - BP_CubSky_01                      昼夜状态来源
  - NS_Env_Common_PlayerVFXDay_LoopCom    白天粒子特效
  - NS_Env_Common_PlayerVFXNight_LoopCom  夜晚粒子特效
  - NS_Env_Snow_01                    雪效果粒子

关键变量 (31个):
  LocationArray, RotationArray, LengthArray  — 区域包围盒数据数组
  ActiveAlong                                — 特效激活标志
  CanOverlay                                 — 是否允许叠加
  System Template                            — Niagara 系统模板
  NiagaraActive                              — 当前激活的 Niagara 组件数组
  BillboardScale                             — 编辑器图标缩放
  RevertRot, EnalePitch                      — 旋转控制
  SplineActor, SplineBoxs                    — 样条引用
  ZOffset, Height                            — 高度参数
  CheckStartInside                           — 初始内部检测标志
  PointsLocation                             — 区域顶点位置数组
  InterSectNum                               — 射线交叉计数
  GrassColor                                 — 草地颜色（含尾随空格）
  ArrowArray                                 — 方向箭头数组
  NiagaraDay, NiagaraNight                   — 昼夜 Niagara 系统引用
  Is Day                                     — 昼夜标志
  TimerClear, AreaEffectActiveAlong,
  TimePlay, TimePause,
  TimerNextPlay, TimerNextPause              — 6个Timer句柄
  bCovered                                   — 遮蔽标志

关键事件/函数 (EventGraph 共 375 个节点):
  Event BeginPlay          — 初始化、启动定时检测

  OnNiagaraActive          — 特效激活回调
  RemoveNiagara            — 移除特效组件
  CustomEnd                — 外部调用结束特效
  CheckScalabilityLevel    — 检测并缓存画质等级

二、优化建议（按优先级排序）| Optimization Suggestions (by Priority)
================================================================================

【高优先级 HIGH】
------------------------------------------------------------------------

1. [逻辑复杂度] EventGraph 节点数量过多（375 个节点），极难维护
   问题：所有逻辑堆积在单一 EventGraph 中，包含大量 Reroute 节点（约180+），
         可读性极差，调试困难，后续开发者几乎无法快速理解意图。
   建议：将逻辑拆分为独立函数（Function），例如：
         - CheckInsideArea()：射线交叉检测逻辑
         - SpawnDayEffect() / SpawnNightEffect()：特效生成
         - HandleDayNightSwitch()：昼夜切换处理
         - CleanupEffects()：特效清理
         - HandleScalability()：画质等级响应
         目标：每个函数不超过 50 个节点，整体 EventGraph 降至 50 节点以内。

2. [命名规范] 多处变量命名存在错误或不规范
   问题：
         - "EnalePitch" → 拼写错误，应为 "EnablePitch"
         - "GrassColor " → 变量名有尾随空格，会导致引用歧义
         - "Is Day" → 布尔变量命名不规范，应遵循 bIsDay 命名约定
         - "NiagaraActive" → 是组件数组，名称未体现复数/数组含义，
           建议改为 "NiagaraActiveComponents"
         - "EnalePitch" 拼写错误可能导致其他开发者无法找到该变量
   建议：统一按 UE 命名规范（布尔用 b 前缀，数组用复数形式）修正全部变量名，
         并全局搜索替换所有引用。

3. [Timer 管理] 存在6个 Timer 句柄，缺乏统一清理保障
   问题：TimerClear、AreaEffectActiveAlong、TimePlay、TimePause、
         TimerNextPlay、TimerNextPause 共6个 Timer，若 Actor 被销毁时
         未在 Event EndPlay 中全部清除，会产生野指针调用崩溃风险。
   建议：确保 Event EndPlay 中对所有 Timer 句柄调用 Clear Timer by Handle，
         并将 Timer 启动/停止逻辑封装在专用函数中统一管理。

------------------------------------------------------------------------
【中优先级 MEDIUM】
------------------------------------------------------------------------

4. [性能] 区域检测基于定时轮询，检测频率需评估
   问题：CheckInside 通过 Set Timer by Event 循环调用，每次执行需遍历
         PointsLocation 数组（O(n)）并对每个样条盒执行向量计算。
         若样条盒数量多且 Timer 间隔过短（如 < 0.1s），在大量 Actor 
         同时存在时会产生较高 CPU 开销。
   建议：
         - 确认 Timer 间隔不低于 0.1-0.2 秒（当前值需核实）。
         - 考虑改用 Overlap 事件（BoxComponent 或 NiagaraVolume）替代
           轮询检测，彻底消除每帧/每tick的检测开销。
         - 如必须保留轮询，在玩家不在附近时（超出最大距离）暂停 Timer。

5. [耦合性] 直接引用 BP_CubSky_01 获取昼夜状态，耦合度高
   问题：Blueprint 直接依赖具体的天空盒 Actor 类来获取 "Is Day" 状态，
         若天空盒系统变更，需同步修改本蓝图。
   建议：通过 GameState、Subsystem 或 Blueprint Interface 暴露昼夜状态，
         BP_AreaCheck 只依赖接口而非具体实现，降低耦合。

6. [编辑器工具] 缺少 Construction Script，编辑器时期无可视反馈
   问题：区域边界仅在运行时通过样条盒计算，编辑器摆放时设计师
         无法直观看到区域范围，容易摆放错误。
   建议：添加 Construction Script，在编辑器模式下绘制调试线框
         （使用 Draw Debug Box / Draw Debug Line），帮助设计师
         直观确认区域范围。

7. [可维护性] 大量 Reroute 节点（约180+个）
   问题：过多的 Reroute 节点是 "意大利面条" 连线的症状，
         严重降低图的可读性。
   建议：将逻辑拆分到函数（见建议1）后，Reroute 数量将自然减少。
         残余的 Reroute 节点应添加注释说明数据含义。

------------------------------------------------------------------------
【低优先级 LOW】
------------------------------------------------------------------------

8. [调试代码] Debug 变量在 Shipping 版本中应被裁剪
   问题：存在 Debug 布尔变量控制调试显示，但若未做平台/版本判断，
         Shipping 版本中仍会保留调试逻辑的执行路径（即使不显示）。
   建议：使用 Development Only 节点或在相关逻辑外包裹 WITH_EDITOR 宏，
         确保调试代码在 Shipping 版本中被完全裁剪。

9. [代码清晰度] bCovered 变量用途不明确
   问题：变量 bCovered 无法从名称推断其具体含义（遮蔽？覆盖？被占用？）。
   建议：重命名为更明确的名称，例如 bIsAreaOccluded 或 bEffectCovered，
         并在变量的工具提示（Tooltip）中添加说明。

10. [组件配置] BillboardComponent 缩放为 5×5×5，可能过大
    问题：默认的编辑器图标缩放5倍，在密集场景中可能遮挡其他 Actor。
    建议：评估是否需要如此大的图标，可通过 BillboardScale 变量动态控制，
          但应将默认值恢复为合理范围（如 1.0 或 2.0）。

11. [变量整理] "System Template" 变量名包含空格
    问题："System Template" 变量名中有空格，虽然 UE 允许，但在 C++ 
          访问或 Python 脚本引用时会造成麻烦。
    建议：重命名为 "SystemTemplate" (PascalCase，无空格)。

12. [CanOverlay 行为] 叠加逻辑需验证防重复生成
    问题：CanOverlay 变量控制是否允许效果叠加，若玩家快速进出区域，
          可能重复触发 SpawnSystemAttached，产生多个叠加的 Niagara 组件。
    建议：在生成前检查 NiagaraActive 数组是否已存在同类组件，
          避免意外的多实例叠加造成性能问题。

三、总结 | Summary
================================================================================
BP_AreaCheck 功能完整，实现了基于样条区域的环境粒子特效管理，支持昼夜切换
和画质分级，具有一定的工程复杂度。

主要问题集中在：
  ① 单一 EventGraph 过于庞大（375节点），维护成本极高 —— 最需优先重构
  ② 多处命名不规范/含错字，影响可读性和引用安全性
  ③ 多 Timer 管理需确保销毁时完整清理，防止崩溃

建议先从【高优先级】问题入手，通过函数化重构和命名修正，大幅降低后续
维护难度，再逐步处理性能和解耦问题。
