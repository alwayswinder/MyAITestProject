一、蓝图概述
--------------------------------------------------------------------------
BP_FlareObject 是一个专门负责 **镜头光晕（Lens Flare）管理** 的 Actor 蓝图，作为整个
LensFlareTool 系统的核心运行时控制器，主要负责以下功能：

1. 在 BeginPlay 阶段初始化所有光晕 Widget（BP_LensFlareWidget），并按 Struct_FlareItem
   数据结构中的配置批量创建光晕元素。
2. 每帧（Event Tick）通过摄像机朝向点积（Dot Product）判断光晕是否在视野内，动态
   开启/关闭各 Widget 的 Tick，以减少屏幕外的无效计算。
3. 与天空盒（BP_CubSky_01）联动：若场景启用了昼夜循环，则绑定 OnSkyTimeUpdate 事件，
   根据当前时间段判断是否显示光晕（TimeRight 标志）。
4. 根据玩家画质设置（Overall Scalability Level）动态启用或禁用光晕效果，确保低配设备
   不受性能拖累（LevelRight 标志）。
5. 将 Actor 自身的世界坐标更新为"摄像机位置 + 太阳方向 × SunDistance"，保证光晕 Widget
   始终朝向太阳（支持可移动模式）。

四、事件与函数执行流程
--------------------------------------------------------------------------

【Event BeginPlay】（节点数量：~30 个节点）
  ├─ 检查 Inited 标志，已初始化则跳过（防重复）
  ├─ 验证 SkyBox 引用有效性（Is Valid 宏 × 2）
  ├─ 检查 SkyBox 的 "Use Time Of Day" 配置
  │    ├─ [True] 绑定 OnSkyTimeUpdate → UpdateFlareActorLocation 事件
  │    └─ [False] 直接初始化
  ├─ 调用 Initiate Flares（批量创建 WidgetList 中的 Widget）
  ├─ 调用 Update Flare Actor Location（定位 Actor 到太阳方向）
  └─ 设置 Inited = True

【Event Tick】（节点数量：~40 个节点，每帧执行）
  ├─ 检查 Inited == True（Branch）
  └─ [True] Sequence：
       ├─ Then_0：可见性判断逻辑
       │    ├─ GetActorLocation（SkyBox）→ 计算太阳方向向量
       │    ├─ GetPlayerController → GetCameraManager（PlayerCameraManager）
       │    ├─ 计算摄像机到太阳方向向量（Normalize）
       │    ├─ Dot Product（摄像机前向 · 太阳方向）> 0 → 得到 bool
       │    ├─ 与 TimeRight、LevelRight、TickWidget 做 AND 布尔运算
       │    ├─ 与 AlwaysTick 做 OR 运算 → EnableTickTmp（本帧值）
       │    └─ 若 EnableTickTmp ≠ 上帧值（Not Equal），则遍历 WidgetList
       │         └─ 对每个 Widget 调用 Set EnableTick
       └─ Then_1：调用 Update Flare Visibility

【Custom Event: UpdateFlareActorLocation】
  ├─ 获取 SkyBox 的世界位置
  ├─ 获取 Actor 自身的前向向量（GetActorForwardVector）
  ├─ 计算目标位置 = SkyBox位置 + 前向量 × SunDistance - 偏移量
  └─ 调用 SetWorldLocation（DefaultSceneRoot）

【Custom Event: CheckScalabilityLevel】
  ├─ 调用 GetGameUserSettings
  ├─ 调用 GetOverallScalabilityLevel → 与阈值比较（integer > integer）
  ├─ 设置 LevelRight 标志（True/False）
  └─ 遍历 WidgetList，根据 LevelRight 设置 Set EnableTick

【Event OnScalabilityModify】
  └─ 调用 Check Scalability Level（响应画质变更）

六、优化建议（按优先级排序）
--------------------------------------------------------------------------

【优先级 P0 - 严重性能风险】
─────────────────────────────────────────────────────────────────────────

1. Tick 内存在多次 ForEach 数组遍历（高开销）
   问题：EventGraph 中存在 6 个 ForEach Loop 宏（MacroInstance_0 ~ MacroInstance_6），
         其中多个位于 Tick 执行路径上（WidgetList 的 Set EnableTick 等），每帧均遍历
         所有 Widget 实例，随 Widget 数量线性增长。
   建议：将"状态未改变时不遍历"的分支前移。当前代码已有 Not Equal 检测，
         但 Update Flare Visibility 每帧调用仍可能触发内部遍历。
         可以考虑将 WidgetList 分批处理（Batch），或将 Update Flare Visibility 改为
         事件驱动（如仅在遮蔽状态变化时调用）。

2. Event Tick 内每帧调用 GetPlayerController + GetCameraManager（频繁对象查找）
   问题：Tick 中每帧通过 GetPlayerController 获取 PlayerController，再获取
         PlayerCameraManager，属于代价较高的对象查找操作。
   建议：在 BeginPlay/Initiate Flares 阶段缓存 PlayerController 和 PlayerCameraManager
         引用到成员变量，Tick 中直接使用缓存变量，避免每帧查找。

【优先级 P1 - 重要逻辑隐患】
─────────────────────────────────────────────────────────────────────────

3. SkyBox 变量被重复读取 6 次（冗余节点）
   问题：WidgetList 中 SkyBox 变量被 6 个独立 VariableGet 节点引用
         （节点 ID: E27F8C66、2C974CF1、053E650E、117E48A0、A97B9799、E8DC1F10），
         对同一帧同一变量重复访存，造成代码冗余且难以维护。
   建议：利用局部变量（Local Variable）或 Sequence 开始处统一读取一次 SkyBox 引用，
         后续节点复用同一引用，减少 VariableGet 节点数量。

4. AlwaysTick 绕过视锥剔除，存在滥用风险
   问题：当 AlwaysTick = True 时，OR 运算会直接使 EnableTick 为 True，完全跳过
         Dot Product 视锥判断，导致屏幕外时仍每帧运行所有 Widget 的 Tick 逻辑。
   建议：在编辑器中为 AlwaysTick 添加详细的 Tooltip 说明其性能影响，或在
         Package/Shipping 构建中强制禁用该标志，仅作为调试辅助选项。

5. Inited 标志依赖顺序执行，缺乏线程安全保障
   问题：BeginPlay 末尾才设置 Inited = True，但若 OnSkyTimeUpdate 事件在
         BeginPlay 执行中途触发（如天空盒已在运行），可能导致
         UpdateFlareActorLocation 在初始化完成前被调用。
   建议：在 BeginPlay 最开始便设置 Inited = True（或使用独立的 LocationInited 标志），
         并在 UpdateFlareActorLocation 中先检查 Inited。

【优先级 P2 - 代码质量改进】
─────────────────────────────────────────────────────────────────────────

6. 大量 Reroute（Knot）节点导致图形可读性差
   问题：EventGraph 中存在 17 个 Reroute 节点（Knot_0 ~ Knot_17 等），说明连线
         走向混乱，图形布局复杂，后续维护难度大。
   建议：将大型 EventGraph 拆分为多个具名函数（如 HandleTickVisibility、
         HandleTimeOfDayUpdate 等），减少单图节点数量（当前 104 个节点），
         用函数调用代替长连线，提升可维护性。

7. Debug 变量在 Shipping 构建中未被编译隔离
   问题：蓝图中 Debug 变量若驱动了屏幕打印或性能消耗逻辑，在 Shipping 包中
         若未手动置为 False，可能带来不必要的开销。
   建议：结合 C++ 侧的 UE_BUILD_SHIPPING 宏，或通过 GameplayTag 条件节点确保
         Debug 逻辑在发布版本中被完全跳过。

8. CheckScalabilityLevel 中 GetGameUserSettings 每次调用均为全局查找
   问题：CheckScalabilityLevel 每次被调用时都执行 GetGameUserSettings，该函数
         内部有一定查找开销。虽然该事件非每帧调用，但仍可优化。
   建议：在 BeginPlay 时缓存 GameUserSettings 引用到成员变量，后续直接使用。

【优先级 P3 - 可维护性建议】
─────────────────────────────────────────────────────────────────────────

9. 缺少函数级注释说明
   问题：EventGraph 中只有 2 个 Comment 节点（EdGraphNode_Comment_0/5/6），
         对关键逻辑区域（如可见性判断逻辑、时间判断逻辑）均无注释说明，
         团队协作时理解成本较高。
   建议：为各功能区域添加 Comment 框，并为 Flare Elements、SunDistance 等
         暴露属性添加 Category 和 Tooltip 描述。

10. Struct_FlareItem 内部字段未在此蓝图中可见文档化
    问题：Flare Elements 数组使用了自定义结构体 Struct_FlareItem，但其内部
          字段（如光晕半径、材质、偏移等）在 BP_FlareObject 图中未有说明。
    建议：为 Struct_FlareItem 中所有字段添加 Tooltip，并在 Flare Elements
          变量上方添加 Comment 说明其数组格式与填写规范。

--------------------------------------------------------------------------
七、总结
--------------------------------------------------------------------------
BP_FlareObject 整体架构设计合理，通过视锥判断+画质分级+时间段检测的三重条件门控
来避免无谓的 Widget Tick，体现了对移动/低配平台的适配考量。

核心待优化点集中在 Tick 路径上的对象查找和数组遍历频率问题（P0），以及代码组织
混乱（大量 Reroute 节点、重复 VariableGet）带来的维护难题（P2）。建议优先处理
P0 和 P1 级问题，以保障游戏在中低配设备上的帧率稳定性。
