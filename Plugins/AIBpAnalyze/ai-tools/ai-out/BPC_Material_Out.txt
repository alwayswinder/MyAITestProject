三、蓝图功能概述
────────────────────────────────────────────────────────────────────────────────
BPC_Material 是一个 **基于曲线驱动的材质参数动画组件**，核心功能如下：

1. 任务注册与管理
   外部代码通过向 MaterialSetting（TMap）中添加 BPS_MaterialSetting 条目来
   注册材质动画任务，支持同时运行多个命名任务（如溶解、高亮、Ghost 等）。

2. Tick 驱动的逐帧更新
   Event Tick 中首先检查 MaterialSetting 是否非空：
     - 若为空 → 通过 Set Component Tick Enabled(false) 关闭 Tick，节省性能。
     - 若非空 → 调用 Material Setting Run 函数驱动所有任务。

3. Material Setting Run 主循环
   每帧对 MaterialSetting 的所有 Key 执行 For Each Loop：
   a. 从 TaskTimeRun 累加 DeltaSeconds，得到当前任务已用时间。
   b. Break BPS_MaterialSetting 解包任务参数。
   c. 【Tag 绑定检查】（注释："是否绑定tag"）
      - 若 BindTag 有效：检查 Owner Actor 是否拥有该 Gameplay Tag。
      - 若 Owner 不含该 Tag → 立即调用 Stop Material Setting 终止任务。
   d. 根据 IsVector 分支：
      - Float 分支：Get Runtime Float Curve Value → On Material Setting F
      - Vector 分支：Get Runtime Vector Curve Value → On Material Setting V
      使用 Map Range Clamped 将任务时间映射到曲线时间范围。
   e. 【AutoEnd 检查】
      - 若当前时间 > TaskLen 且 AutoEnd=true：
        * 调用 Stop Material Setting（停止任务）
        * 调用 Remove Material Setting（从 Map 中移除条目）
        * 注释说明："只移除map，不重置材质"（材质参数保持在最终值）。

4. MaterialSettingRun 自定义事件
   外部可通过触发此事件来手动触发一次 Material Setting Run，便于在特定
   时机强制刷新（如刚注册任务后立即触发一帧更新）。

5. Begin Play 初始化
   禁用组件 Tick（Set Component Tick Enabled = false），等待任务注册后
   再按需开启，避免空帧消耗。

四、优化建议（按优先级排序）
────────────────────────────────────────────────────────────────────────────────

【P0 - 严重性能风险】
────────────────────
▶ 问题1：Tick 内每帧调用 Map.Keys()，产生数组拷贝与堆分配
  位置：Get MaterialSetting → Keys → For Each Loop（节点 128814FB）
  原因：TMap.Keys() 每次调用都会产生一个新的 TArray 拷贝，在高帧率（60+）
        和多任务（N 个 Key）场景下，每帧产生 N 次查找 + 1 次 Keys 拷贝，
        GC 压力显著。
  建议：维护一个独立的 TArray<FName> ActiveKeys 与 MaterialSetting 同步
        增删，避免每帧调用 Keys()；或改用 TArray<FName> 作主容器。

▶ 问题2：For Each 内对同一 Key 执行多次 Map.Find()
  位置：Find（TaskTimeRun）、Find（MaterialSetting）、Find（多个）在同一
        循环迭代中重复出现（节点 6D5B8F6A、7AA08938、EE4A565E、200D73C0 等）
  原因：每次 Find 均为 O(1) 但带缓存不命中开销；在循环体内对同一 Key 的
        重复查找可合并为一次，减少开销。
  建议：在循环体顶部一次性获取所需所有值，将中间结果赋给局部变量。

【P1 - 架构与可维护性问题】
────────────────────────────
▶ 问题3：Array ElementTmp 作为组件级成员变量
  位置：变量 "Array ElementTmp"（多处 Get/Set，如节点 C705B80241、
        BD3D020848、A149F53C4E 等）
  原因：将 For Each 循环的迭代临时量定义为组件成员变量，存在以下风险：
        - 不同调用栈或延迟调用时变量被覆盖，引发难以复现的 Bug。
        - 语义上属于局部变量，暴露在组件作用域违反封装原则。
  建议：将其改为函数内的局部变量（Local Variable），或直接使用 For Each
        Loop 节点输出的 "Array Element" 引脚，无需中间变量。

▶ 问题4：Delta Seconds 保存为组件变量
  位置：Set Delta Seconds（节点 26A9D92B）→ Get Delta Seconds（节点 2A677834）
  原因：DeltaSeconds 应在函数调用链中以参数形式传递，而非通过组件成员
        保存再读取，这破坏了函数纯洁性并增加了理解成本。
  建议：在 Material Setting Run 函数签名中加入 float DeltaSeconds 入参，
        由 Event Tick 直接传入。

▶ 问题5：Ghost 残影相关变量与通用材质系统耦合
  位置：GhostTrailTimer、Time Disappear、M_Ins_Ghost、M Name_Ghost、
        M Curve_Ghost（均为组件级变量）
  原因：Ghost 特效是一种具体业务逻辑，与通用材质动画系统（MaterialSetting）
        混合在同一组件中，违反单一职责原则，且这些变量目前未在 EventGraph
        中被使用（疑似遗留代码或由外部逻辑直接操作）。
  建议：将 Ghost 逻辑独立为子组件（BPC_GhostTrail）或单独函数库，
        或明确这些变量的调用位置并添加注释。

【P2 - 代码质量与可读性】
────────────────────────────
▶ 问题6：过多的 Reroute（转接）节点（12+ 个）
  位置：K2Node_Knot_68、74、103~112 等
  说明：大量转接节点表明图表布局混乱、逻辑路径交叉。
  建议：将主循环体内的复杂逻辑重构为命名函数（如 ProcessSingleMaterialTask、
        EvaluateMaterialCurve），提升可读性并减少连线交叉。

▶ 问题7：注释覆盖率极低
  现状：72 个节点中仅有 2 个注释节点（"是否绑定tag"、"只移除map，不重置材质"），
        其余关键逻辑节点均无注释说明。
  建议：为每个主要功能区域（Init、Tag 检查、曲线求值、自动终止）添加
        Comment 节点标注，便于团队维护。

▶ 问题8：变量命名不一致、语义不明
  问题变量：
    - "Use To"：名称过于泛化，无法表达实际用途
    - "M Name V" / "M Curve V"：命名过于简短，含义模糊
    - "MaterialSave"：未注明保存的目的（备份？回调？）
  建议：采用统一命名规范，如：
    - Use To → MaterialApplyContext
    - M Name V → VectorParamName
    - M Curve V → VectorParamCurve
    - MaterialSave → OriginalMaterialBackup

▶ 问题9：AutoEnd 逻辑判断位置
  位置：float > float（节点 42B28F92）后的 Branch（节点 E3462F71）
  说明：AutoEnd 的终止判断发生在本帧材质参数已更新之后，这意味着最后
        一帧的曲线终值可能超出 TaskLen 对应的位置（取决于 DeltaTime）。
  建议：在 Map Range Clamped 的 InRangeB 处使用 Min(CurrentTime, TaskLen)
        限制最终采样时间，确保曲线求值不超出设计范围。

【P3 - 次要改进点】
────────────────────
▶ 问题10：Begin Play 中仅关闭 Tick，无其他初始化逻辑
  建议：在 Begin Play 阶段添加对 MainMesh 的有效性验证（IsValid），
        防止后续运行时空指针错误。

▶ 问题11：Remove Material Setting 注释说明"不重置材质"
  说明：这是有意设计，但可能造成任务结束后材质参数停留在最终值，与
        调用者的预期不符（可能需要归零或恢复原始值）。
  建议：为 StopMaterialSetting 提供可选的 bool bResetToDefault 参数，
        让调用方自行决定是否重置材质参数至初始值。

五、总结
────────────────────────────────────────────────────────────────────────────────
BPC_Material 是一个设计思路清晰的 **曲线驱动材质动画组件**，支持多任务
并发、GameplayTag 条件绑定和自动结束，具备一定的通用性。

主要问题集中在：
  1. Tick 内存在逐帧的容器分配与重复查找（P0 性能瓶颈）
  2. 临时变量和 DeltaSeconds 的作用域设计不规范（P1 隐患）
  3. Ghost 残影逻辑与通用系统耦合，职责不清晰（P1 可维护性）
  4. 图表节点过多、注释缺失，长期维护成本高（P2 可读性）

优先处理 P0 问题（Keys 拷贝、重复 Find），可显著降低高频 Tick 开销；
P1 问题（临时变量规范化、Ghost 解耦）有助于消除潜在的逻辑 Bug。
