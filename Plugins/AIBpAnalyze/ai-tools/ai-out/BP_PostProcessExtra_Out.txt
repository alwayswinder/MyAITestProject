一、蓝图概述
--------------------------------------------------------------------------
BP_PostProcessExtra 是一个基于区域触发的 **天空/光照/后处理效果区域过渡控制器**。
其核心职责是：当玩家进入或离开指定区域（触发体）时，以平滑插值的方式切换
天空球、太阳光颜色、本地雾效等大气参数，从而实现不同地图区域之间的视觉风格
无缝过渡（例如室外→室内→特殊区域的天气/光照变化）。

主要功能模块：
1. 触发检测       — BrushComponent 碰撞体，监听 BeginOverlap / EndOverlap
2. 光照过渡       — 使用 Timeline + Lerp(LinearColor) 对四段时段光色进行插值
3. 天空纹理更新   — 通过 BP_CubSky_01 的 Init/Update Sky Tex/Effects 接口驱动天空材质
4. 本地雾效管理   — 引用 BP_Locality_fog，保存/恢复四段时段雾参数
5. 后处理标志管理 — EnableSunLight / EnableSkyTex 两个开关控制子系统开启
6. 状态持久化     — 进入/离开区域前后分别保存原始光色与雾参数，供恢复使用
7. UV 偏移管理    — UVoffset / UVOffset_Save 控制天空纹理平铺与滚动
8. 编辑器预览     — Preview 自定义事件支持编辑器内效果预览

四、事件与执行流程
--------------------------------------------------------------------------
1. Event BeginPlay
   └─ Is Valid(PostActorIn)
      ├─ [Valid] 绑定 BrushComponent 的 OnBeginOverlap → OverlapStart
      │          绑定 BrushComponent 的 OnEndOverlap   → OverlapEnd
      └─ 执行 Sequence → 初始化天空纹理数组（清空 Editor 数组）→ 检查 EnableSunLight/EnableSkyTex

2. OverlapStart（玩家进入触发体）
   └─ Equal(OtherActor, GetPlayerCharacter) 判断是否为玩家
      └─ [True] 3路 Sequence:
          then_0 → 保存当前光照颜色快照（四段时段 LightSave）
          then_1 → 启动 Timeline，逐帧通过 Lerp(LinearColor) 混合四段光色
                   → Timeline Update → Update Sun Color(Night/Dust/Day/Morning)
          then_2 → Init Sky Effects → Update Sky Tex → Update Sky Effects (on BpSky)
                   → 同步更新本地雾参数（四段 Fog Save）

3. OverlapEnd（玩家离开触发体）
   └─ Equal(OtherActor, GetPlayerCharacter) 判断是否为玩家
      └─ [True] 反向播放 Timeline（ReverseFromEnd），将光色/雾效恢复到进入前快照

4. Preview（自定义事件，编辑器调用）
   └─ 不依赖运行时触发，直接执行 Init/Update Sky Effects 以便编辑器内预览效果

5. Timeline
   └─ NewTrack_0（0→1 曲线输出 Alpha）
      → Lerp(LinearColor) × 4（每段时段各一组 Lerp）
      → Update Sun Color（将插值后颜色写入 BP_CubSky_01 的光源参数）

六、优化建议（按优先级排序）
--------------------------------------------------------------------------

【P0 — 严重 / 影响维护性与稳定性】

1. 事件图表节点数过多（277节点），且大量使用 Reroute 节点（目测 50+ 个）
   问题：单一 EventGraph 承载全部逻辑，导致图表极度复杂，调试困难，协作冲突风险高。
   建议：将以下逻辑重构为独立函数（Function）：
     - SaveLightState()   — 保存四段光色快照
     - RestoreLightState()— 恢复快照
     - TransitionSkyTo()  — 天空过渡统一入口（封装 Init/Update 调用链）
     - SaveFogState() / RestoreFogState()
   预期收益：节点数可降低至 60~80 个，消除大部分 Reroute 线缆。

2. 玩家校验使用 Equal(Object) 而非标签/接口过滤
   问题：OverlapStart/End 均通过 GetPlayerCharacter() == OtherActor 判断，
         在多人游戏或本地分屏场景下会产生误判（只响应 Player 0）。
   建议：改用 Cast<APlayerCharacter> 或检查 Actor 的 Tag，或通过
         碰撞通道（Overlap 只响应 PlayerPawn 通道）在物理层面过滤，
         去除蓝图层的逐帧对象比较。

【P1 — 高优先级 / 影响性能与扩展性】

3. 状态变量结构散乱，存在大量平行 *Save 变量
   问题：Night LightSave / Dust LightSave / Day LightSave / Morning LightSave
         + Night Fog Save / Dust Fog Save / Day Fog Save / Morning Fog Save
         共 8 个平行变量，每次修改时段数量都需要成倍增加变量。
   建议：定义 FTimeOfDaySkyState 结构体（包含 Light Color × 4 + FogParams × 4），
         用单一 Save 变量替换 8 个分散变量，并便于序列化/持久化。

4. 数组每次都全量 Clear 重建（SkyTexArray_Editor / SkyMaskArray_Editor）
   问题：BeginPlay 和 Preview 中均执行 Clear + 重新填充数组，
         若数组较大或频繁调用 Preview，会产生不必要的 GC 压力。
   建议：增加脏标记（bSkyArrayDirty），仅在数据发生变化时重建数组；
         Editor 专用数组与运行时数组分离管理，避免运行时访问编辑器资源。

5. BP_CubSky_01 硬引用耦合
   问题：BpSky 变量直接持有 BP_CubSky_01 实例引用，两个蓝图紧耦合，
         无法在不同关卡中替换天空系统实现。
   建议：为天空系统定义 ISkyController 接口（包含 InitSkyEffects、
         UpdateSkyTex、UpdateSkyEffects、UpdateSunColor 方法），
         BpSky 变量类型改为接口引用，解耦具体实现。

【P2 — 中优先级 / 代码质量与可读性】

6. CollisionStartTimer 的使用暗示 BeginPlay 时序问题
   问题：使用 Timer 延迟处理碰撞绑定，说明依赖的组件或外部对象在 BeginPlay 时
         尚未就绪，这是一种脆弱的时序 hack，在不同硬件上延迟可能不稳定。
   建议：改为订阅游戏模式的 OnLevelReady 或使用 Actor 的 PostInitializeComponents
         / PostBeginPlay 事件；或通过 GameMode/GameInstance 统一管理初始化顺序。

7. EnableSunLight / EnableSkyTex 两个独立开关逻辑分支重复
   问题：两个 Branch 节点导致相似逻辑执行两条几乎平行的分支，增加节点数量。
   建议：合并为位掩码枚举（ESkyUpdateFlags）或将两个 bool 的组合逻辑提取为
         函数 bool ShouldUpdateSky()，减少分支节点重复。

8. 大量 Reroute（Knot）节点（K2Node_Knot_500 ~ K2Node_Knot_562）
   问题：超过 60 个 Reroute 节点说明连线跨越距离极长，图表布局混乱。
   建议：通过上述函数拆分（见 P0-1）从根本上缩短连线距离；同时为关键变量
         使用局部变量（Local Variable）或在函数内部传参，避免跨区域长线。

【P3 — 低优先级 / 维护友好性】

9. 缺少节点注释
   问题：277 个节点中几乎没有注释（nodeComment 均为空），
         新接手的开发者很难理解各段逻辑的意图。
   建议：为每个主要逻辑块添加 Comment Box，至少标注：
         触发绑定区、状态保存区、Timeline 插值区、天空初始化区、雾效更新区。

10. 编辑器专用数组（_Editor 后缀）与运行时数组职责不清
    问题：SkyTexArray_Editor 与 SkyTexArray 并存，未通过 #if WITH_EDITOR 宏
          或条件编译隔离，运行时包体中仍包含编辑器专用数据。
    建议：将 Editor 专用变量标记为 EditorOnly，或在打包时通过开发工具剥离，
          避免增大运行时内存占用。

--------------------------------------------------------------------------
七、总结
--------------------------------------------------------------------------
BP_PostProcessExtra 设计思路清晰——区域触发 + 状态保存/恢复 + 平滑插值过渡，
是一个功能完备的局部天空/后处理切换方案。

核心问题集中在 **可维护性** 上：
  - 单一 EventGraph 承载过多逻辑（277节点）是最紧迫的技术债务；
  - 大量 Reroute 节点是上述问题的直接症状；
  - 状态变量缺乏结构化导致未来扩展成本线性增长。

优先完成 P0 级重构（拆函数 + 清理 Reroute）后，
再推进 P1 级解耦工作（接口化 + 结构体化状态），
即可将该蓝图的可维护性提升到与项目规模相匹配的水平。
