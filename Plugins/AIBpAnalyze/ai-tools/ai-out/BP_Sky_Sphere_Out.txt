一、蓝图功能概述 (Blueprint Functionality Overview)
================================================================================

BP_Sky_Sphere 是 Unreal Engine 的内置天空球 Actor，核心职责是：
  1. 展示一个超大球体（Scale 400×400×400）作为天空背景
  2. 通过动态材质实例驱动天空颜色、云、星星等视觉效果
  3. 读取场景中的方向光（Directional Light Actor）旋转角度，计算太阳高度，
     并据此动态更新天空颜色（支持昼夜渐变曲线或固定手动颜色两种模式）

二、优化建议（按优先级排序）
   Optimization Suggestions (by Priority)
================================================================================

────────────────────────────────────────────────────────────────────────────────
[P0 - 高优先级 / Critical] 问题：EventGraph 为空，蓝图无运行时自动更新能力
────────────────────────────────────────────────────────────────────────────────
  现状：
    EventGraph 中没有任何节点（0 nodes）。蓝图不响应 BeginPlay、不 Tick，
    也没有 Construction Script（未检测到）。这意味着：
    · 若方向光运行时旋转变化，天空不会自动跟随更新
    · 用户必须手动调用 RefreshMaterial / UpdateSunDirection，
      或通过其他蓝图外部驱动

  风险：
    在动态昼夜系统中，天空颜色不随时间变化，造成画面与光照脱节

  建议：
    方案A（低性能开销）：添加 Event BeginPlay → 调用 RefreshMaterial；
      添加 Event Tick（以低频率，如 0.5s 间隔）→ 调用 UpdateSunDirection
      使用 Set Timer by Function Name 替代 Tick 可进一步降低开销
    方案B（编辑器友好）：添加 Construction Script，在 Details 面板参数变更时
      自动刷新，改善关卡编辑体验

────────────────────────────────────────────────────────────────────────────────
[P1 - 高优先级 / High] 问题：RefreshMaterial 与 UpdateSunDirection 大量逻辑重复
────────────────────────────────────────────────────────────────────────────────
  现状：
    两个函数均包含：
    · 从 Directional Light 获取旋转 → 计算 Sun Height
    · Set Vector Parameter Value（太阳方向、太阳颜色）
    · Absolute(Float) + Lerp + float < float + Select Float 等太阳位置计算链
    RefreshMaterial (76节点) ≈ UpdateSunDirection (38节点) + 额外颜色/云/星星参数

  风险：
    · 维护成本高：修改太阳高度计算逻辑需在两处同步
    · 代码膨胀：RefreshMaterial 节点数量庞大，阅读和调试困难

  建议：
    将共同的"太阳高度计算 + 太阳方向设置"提取为独立的私有函数
    （如 ComputeSunHeight、SetSunDirectionParameters），
    RefreshMaterial 和 UpdateSunDirection 均调用该子函数，
    消除 ~20 个重复节点

────────────────────────────────────────────────────────────────────────────────
[P1 - 高优先级 / High] 问题：RefreshMaterial 函数过于庞大（76节点），可读性差
────────────────────────────────────────────────────────────────────────────────
  现状：
    76 个节点在同一函数图中平铺，包含颜色分支、云参数、星星亮度、
    光照颜色读取等多个功能域

  建议：
    按功能域拆分为子函数：
    · SetSunParameters()     — 太阳方向、亮度
    · SetSkyColorParameters() — 天顶/地平线颜色（含曲线/手动模式分支）
    · SetCloudParameters()   — 云速、透明度、颜色
    · SetStarParameters()    — 星星亮度
    主函数 RefreshMaterial 仅按顺序调用以上4个子函数

────────────────────────────────────────────────────────────────────────────────
[P2 - 中优先级 / Medium] 问题：缺少脏标记（Dirty Flag）机制，每帧全量设置参数
────────────────────────────────────────────────────────────────────────────────
  现状：
    无论参数是否改变，每次调用都会向 Dynamic Material Instance 发送
    全量 SetScalarParameterValue / SetVectorParameterValue 调用（约 10+ 次）

  风险：
    在 Tick 或高频驱动场景下，造成不必要的材质参数设置开销

  建议：
    添加缓存变量（如 LastSunHeight、LastDirectionalLightRotation），
    每次更新前比较变化幅度，若差值小于阈值（如 0.001f）则跳过该帧更新

────────────────────────────────────────────────────────────────────────────────
[P2 - 中优先级 / Medium] 问题：Directional Light Actor 引用无类型约束且多次重复读取
────────────────────────────────────────────────────────────────────────────────
  现状：
    变量 "Directional light actor" 类型为 Actor，而非更精确的 DirectionalLight。
    在 RefreshMaterial 中，该引用被读取 6 次（Get Directional light actor × 6），
    每次都通过 Get Actor Rotation / Get Rotation X Vector 单独访问

  建议：
    · 将变量类型改为 DirectionalLight（更精确，可直接访问 LightComponent）
    · 在函数开头一次性读取旋转和方向，缓存到局部变量，
      避免重复 Get Actor Rotation 调用

────────────────────────────────────────────────────────────────────────────────
[P2 - 中优先级 / Medium] 问题：Stars brightness 通过 float + float 节点计算，逻辑不透明
────────────────────────────────────────────────────────────────────────────────
  现状：
    在 RefreshMaterial 中，Stars brightness 通过
    "K2Node_CommutativeAssociativeBinaryOperator_995 (float + float)"
    与其他数值相加后传入材质，缺乏注释说明加法意图

  建议：
    添加节点注释说明计算公式（如：Stars = StarsBrightness + NightOffset），
    或提取为独立 Pure Function GetEffectiveStarsBrightness()，
    使意图自文档化

────────────────────────────────────────────────────────────────────────────────
[P3 - 低优先级 / Low] 问题：颜色模式切换（Colors determined by sun position）无回调通知
────────────────────────────────────────────────────────────────────────────────
  现状：
    颜色模式开关是普通布尔变量，切换后需手动调用 RefreshMaterial 才能生效

  建议：
    为该变量添加 OnRep 或使用 Property Setter 函数，
    在值改变时自动触发 RefreshMaterial，提升使用易用性

────────────────────────────────────────────────────────────────────────────────
[P3 - 低优先级 / Low] 问题：无法在编辑器中预览昼夜变化
────────────────────────────────────────────────────────────────────────────────
  现状：
    没有 Construction Script，编辑器中调整参数不会实时反馈

  建议：
    添加 Construction Script（调用 RefreshMaterial），
    使 Details 面板参数调整后立即在视口中预览效果

三、总结 (Summary)
================================================================================

  BP_Sky_Sphere 是一个功能完整的天空球实现，依赖曲线资产实现昼夜颜色渐变，
  架构清晰。主要问题集中在：

  1. 缺乏自驱动（无 Tick/BeginPlay）→ 需外部调用才能更新，适用性受限
  2. RefreshMaterial 和 UpdateSunDirection 存在大量重复代码
  3. RefreshMaterial 函数过于臃肿，建议按功能域拆分

  修复 P0/P1 问题可显著提升运行时正确性与代码可维护性，
  修复 P2 问题可优化在动态昼夜场景下的性能表现。
